What it is, how it's built, what's included, and what a buyer gets.

The problem it solves

Most systems that process human-written material either bake domain assumptions into the model layer — through fine-tuning, prompt engineering that mixes domain logic with reasoning, or separate models per domain — or they produce generic output that doesn't hold up under real-world complexity.

None of them handle the specific problem of reading unstructured longitudinal input — material written over time, in fragments, from a person under load — with the epistemological honesty that real-world consequential use requires.

The core insight: separate how to read from what to read for. The reading primitives are domain-agnostic. The domain knowledge lives in a thin steering layer. The two compose at runtime without either knowing about the other.

Three-layer architecture

The stack has three distinct layers with clean contracts between them. This is the architectural decision that makes the whole thing work.

Layer 1 — Behaviours
16 behaviour modules
Domain-agnostic reading primitives. Each behaviour asks a single epistemological question: is this movement or stuck, is this avoidance or constraint, is this confidence supported or performed, what connects across time. Behaviours have no domain knowledge. They run identically across every sector.
Layer 2 — Guides
22 sector guides
Domain knowledge as steering rules. Each guide defines what movement looks like in this sector, what avoidance looks like here, what confidence requires here, which signals are load vs choice. Guides contain no engine logic. They steer — they don't run.
Layer 3 — Components
12 adaptive components
Upper-layer response modules that act on a completed map. Report writers, handover generators, consent gates, confidence-gated output. Components never touch the reading layer — they receive a map and produce output appropriate for the described audience and context.

The spine composes them. A behaviour reads the material using the guide's steering rules without importing the guide. The guide calibrates the behaviour without containing any reading logic. This means the same 16 behaviours read a reentry case, a legal brief, a grief map, and a game narrative — and produce correctly calibrated reads for each — without any behaviour knowing which domain it's in.

The 16 behaviours

Each behaviour is a self-contained module that reads one aspect of the input. They run independently and their signals combine in the map.

  • Independent signal counting — what appears across sources without coordination
  • Baseline vs live material — distinguishing the starting state from movement
  • Correction as governance — user corrections override and propagate forward
  • Avoidance detection — distinguishing avoidance from genuine constraint
  • Load-sensitive capability — what becomes harder under high load
  • Contradiction holding — surfacing tension without resolving it
  • Open gap discipline — what's absent and why it matters
  • Connections across time — what links across entries
  • Movement vs non-movement reading — what is and isn't actual progress
  • Next useful move — what the map suggests without prescribing

Key mechanisms worth knowing

Confidence calibration

Six levels: not_readable → inferred → thin → partial → supported → strong. Each claim in the output carries a confidence label derived from how many independent signals support it, not from a model's self-assessment. A highStakes gate suppresses output below a threshold in high-risk sectors. The system can say "I cannot read this reliably enough" — structurally, not as a disclaimer.

Correction as governance

User corrections are first-class signals, not overrides. When a user says the map has misread something, that correction propagates forward and changes how subsequent material is weighted. The user's authority over their own material is built into the reading layer, not bolted on.

Simulation isolation

The gaming and simulation components (hidden agenda, faction memory, dramatic timing, trust drift) cannot activate in clinical, legal, or real-world deployments. The spine rejects them at attachment unless the operator explicitly sets allowSimulationComponents: true. This is a hard architectural boundary, not a policy note in a README.

The assembly pipeline

A guide file is a plain JS object. Running the assembler against it produces a complete single-file deployable HTML application — no server, no build step, no dependencies, works offline.

Guide (.js) → assemble.js → Cartridge → Single HTML file → Deploy anywhere

Adding a new sector means writing one guide file and running one command. The reading layer requires no changes.

What this is not

This is not a wrapper around a language model. The stack does not call any AI API. It is a deterministic reading engine — pattern matching, signal counting, confidence calculation — that runs entirely in the client. No model, no inference cost, no latency, no hallucination risk.

That is both a constraint and a strength. The constraint: it reads what is written, not what might be meant. The strength: the output is fully deterministic, fully auditable, fully explainable, and costs nothing to run at scale.

The position

Most AI system design doesn't separate how to read from what to read for. The domain knowledge and the reasoning logic get coupled — in the model, in the prompt, in the fine-tune. When you need to add a domain, you rebuild the reading layer. When the reading layer needs improving, you risk breaking domain knowledge that was embedded in it.

This stack solves that problem with a clean three-layer separation. The reading layer is stable and domain-agnostic. The domain knowledge is a thin, auditable, replaceable steering file. The composition is the spine's job. None of the three layers needs to know about the other two.

That separation is the asset. Everything else — the 22 sectors, the 17 assembled maps, the report writers, the confidence system — is built on top of it and is therefore extensible in the same way.

A buyer who understands the architecture can add a sector in a day, deploy a map in minutes, and extend the reading layer without touching the domain knowledge they've already built.
